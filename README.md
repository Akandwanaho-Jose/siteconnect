# SiteConnect Uganda

SiteConnect Uganda is a Flutter + Firebase mobile application for public construction project coordination and monitoring in Uganda.

## Flutter Setup

```bash
flutter pub get
flutter analyze
flutter test
```

Firebase platform configuration is generated by FlutterFire. Do not edit `lib/firebase_options.dart` manually.

## Firebase Seed Setup

The seed tooling uses the Firebase Admin SDK and is intentionally idempotent: it checks each starter document first and never overwrites existing Firestore data.

### 1. Install Node Dependencies

```bash
npm install
```

### 2. Prepare Credentials

Use one of these credential options:

```powershell
# Option A: ignored service account file
$env:FIREBASE_SERVICE_ACCOUNT_FILE="tools/seed/credentials/service-account.json"

# Option B: Google Application Default Credentials
$env:GOOGLE_APPLICATION_CREDENTIALS="C:\path\to\service-account.json"

# Option C: JSON string in environment
$env:FIREBASE_SERVICE_ACCOUNT_JSON="{...}"
```

Recommended local layout:

```text
tools/seed/credentials/service-account.json
```

That folder is ignored by Git. Never commit service account keys, `.env`, or other secret files.
`.env.example` is provided as a template; the seed script reads variables from your shell environment.

### 3. Choose the First Admin

Create or identify a Firebase Authentication user first. The seed script creates only the Firestore profile for that Auth user.

```powershell
$env:SEED_ADMIN_EMAIL="admin@example.com"
$env:SEED_ADMIN_NAME="SiteConnect Administrator"
$env:SEED_ADMIN_DISTRICT="Kampala"
```

You may use `SEED_ADMIN_UID` instead of `SEED_ADMIN_EMAIL`.

### 4. Check the Seed Plan

```bash
npm run seed:check
```

This verifies credentials and prints the documents that would be created.

### 5. Run the Seed

```bash
npm run seed
```

The seed creates starter documents for:

- `roles`
- `system_settings`
- one sample `projects` record
- sample `project_members`
- the first admin `users/{uid}` profile, only if missing

Existing documents are skipped.

### Optional Role Test Users

For development and QA, the seed can create one Firebase Authentication user
and matching Firestore `users/{uid}` profile for every supported role.

```powershell
$env:SEED_TEST_USERS="true"
$env:SEED_TEST_USERS_PASSWORD="SiteConnect@12345"
$env:SEED_TEST_USER_EMAIL_DOMAIN="siteconnect.test"
$env:SEED_TEST_USER_EMAIL_PREFIX="siteconnect"

npm run seed:check
npm run seed
```

Default test logins:

- `siteconnect.administrator@siteconnect.test`
- `siteconnect.project-manager@siteconnect.test`
- `siteconnect.site-engineer@siteconnect.test`
- `siteconnect.contractor@siteconnect.test`
- `siteconnect.consultant@siteconnect.test`
- `siteconnect.clerk-of-works@siteconnect.test`
- `siteconnect.district-engineer@siteconnect.test`
- `siteconnect.procurement-officer@siteconnect.test`
- `siteconnect.community-representative@siteconnect.test`
- `siteconnect.environment-officer@siteconnect.test`
- `siteconnect.community-development-officer@siteconnect.test`

All of the default test accounts use the password configured by
`SEED_TEST_USERS_PASSWORD`. These accounts are for non-production testing only.

## Local User Administration

Administrators can manage users in the app from
`Dashboard > Administration > Users and roles`.

On the Firebase Spark/free plan, SiteConnect changes an existing login email by
creating a replacement login with the new email and deactivating the old profile
inside the app. The old Firebase Auth account still exists in Firebase, but it
cannot open SiteConnect because its Firestore profile is inactive.

Use this local Admin SDK tool only as a fallback or for bulk maintenance.

Prepare credentials:

```powershell
$env:FIREBASE_SERVICE_ACCOUNT_FILE="tools/seed/credentials/service-account.json"
```

List Firebase Auth users with their Firestore roles:

```bash
npm run users:list
```

Create a real login account and matching Firestore profile:

```powershell
npm run users:create -- `
  --email jane.nalwoga@example.com `
  --password TempPass123 `
  --name "Jane Nalwoga" `
  --role site_engineer `
  --district Kampala
```

Change an existing user's real login email in Firebase Auth and Firestore:

```powershell
npm run users:update-email -- `
  --from siteconnect.site-engineer@siteconnect.test `
  --to jane.nalwoga@example.com
```

Change a user's role:

```powershell
npm run users:update-role -- `
  --email jane.nalwoga@example.com `
  --role project_manager
```

To update many test emails in one pass, copy
`tools/admin/email-map.example.json` to `tools/admin/email-map.local.json`,
replace the `to` values with real emails, then run:

```powershell
npm run users:apply-email-map -- --file tools/admin/email-map.local.json
```

## Passwords

Signed-in users can change their own password from
`Edit profile > Change password`. They must enter their current password first.

Users who forget their password can use `Forgot password?` on the login screen.
Firebase sends a reset link to the account email. This requires a real inbox;
test emails such as `@siteconnect.test` cannot receive reset emails.

## Firestore Rules and Indexes

Rules and indexes are tracked in:

- `firestore.rules`
- `firestore.indexes.json`

Deploy them with:

```bash
firebase deploy --only firestore:rules,firestore:indexes
```

The current rules are simple and safe for MVP development: authenticated users can read core operational records, admin/project-manager roles can manage protected records, and users can only read their own notification/profile data where appropriate.
